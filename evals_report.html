⚠️ Section 2 — Failure Modes
  • Context Drift: In long multi-turn conversations (>10 turns), the model occasionally over-generalises Lily's skills beyond the knowledge base.
  • Tool Latency: GitHub pagination and Cal.com availability calls add 1–3 s of perceived lag, causing awkward silences in voice mode.
  • Ambiguous Queries: Vague prompts like "tell me about her projects" sometimes retrieve only the most recent KB chunk instead of synthesising all projects.
  • Partial Booking Flow: createBooking sometimes fails silently if Cal.com returns a non-200 status; no user-facing error is surfaced.
  • Voice-Text Parity Gap: The ElevenLabs voice agent and text chat share different system prompts, causing minor inconsistencies in tone and scope.
⚖️ Section 3 — Design Tradeoffs

✅ Strengths

  • Groq + Llama 3.3 70B gives near-zero inference latency vs. GPT-4o at a fraction of cost.
  • Static RAG (Markdown KB) is fast to load and version-control; no DB round-trips on every message.
  • ElevenLabs streaming enables real-time voice with natural prosody, differentiating from basic TTS.
  • Server Actions for tools keep secrets server-side; no API keys exposed to the client.

⚠️ Limitations

  • No vector DB: Semantic search is approximated by full-context injection; scales poorly with large KBs.
  • Stateless sessions: No persistent chat history across page reloads; context window is the only memory.
  • Dual-agent complexity: Maintaining two separate agents (text + voice) doubles prompt-engineering overhead.
  • Cal.com dependency: Scheduling fails entirely if Cal.com API is down — no graceful fallback.
🚀 Section 4 — Future Improvements
  • Supabase pgvector RAG: Replace static MD injection with semantic vector retrieval to improve accuracy at scale and reduce token usage by ~60%.
  • Unified Agent Architecture: Merge text and voice into one shared system prompt + tool registry to eliminate parity gaps and reduce maintenance cost.
  • Streaming Tool UX: Show live "Checking GitHub…" / "Fetching availability…" skeleton cards during tool calls to eliminate perceived latency in the UI.
  • Persistent Memory: Add a Supabase session store so the AI retains cross-session context (e.g., "last time you asked about X…") for returning recruiters.
  • Automated Eval Pipeline: Implement a nightly CI eval harness using the recruiter checklist queries with LLM-as-judge scoring to catch regressions before deployment.
  • Booking Resilience: Add retry logic and user-facing error states for Cal.com failures; cache last-known availability slots as a graceful fallback.
